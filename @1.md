# TeX Machina 개선 제안 사항 (@fix.md)

이 문서는 `tex-machina` VS Code 확장 프로그램의 코드베이스 분석을 바탕으로, 아키텍처, 안정성, 유지보수성 및 성능 측면에서 개선이 필요한 사항들을 정리한 것입니다.

## 1. 아키텍처 및 코드 구조 개선 (Architecture)

### 1.1 `extension.ts` 리팩토링 (God Object 분리)
- **현황**: 현재 `extension.ts`가 Python 프로세스 생명주기 관리, 명령어 등록, UI 업데이트, 설정 로드 등 너무 많은 책임을 가지고 있습니다.
- **개선**: 기능을 전문화된 서비스 클래스로 분리하십시오.
    - `PythonBridge`: Python 프로세스 실행, `stdin/stdout` 통신, 재시작 로직 담당.
    - `CommandRegistry`: 확장 프로그램 명령(`registerCommand`) 관리.
    - `StatusBarManager`: 상태 표시줄 UI 업데이트 전담.

### 1.2 Webview Modernization
- **현황**: `webviewProvider.ts` 내부에 거대한 HTML/JS/CSS 문자열이 포함되어 있어 린팅, 타입 체크, 컴포넌트화가 어렵습니다.
- **개선**: Webview 코드를 별도의 프론트엔드 프로젝트(예: Svelte, React, 또는 순수 TS/SCSS)로 분리하고, `esbuild` 등을 사용하여 번들링된 파일을 로드하도록 변경하십시오.

## 2. TypeScript - Python 통합 및 통신 (Bridge & Protocol)

### 2.1 통신 프로토콜 정형화 (Schema-based JSON-RPC)
- **현황**: 현재 단순 JSON 문자열을 `stdin/stdout`으로 주고받으며, 데이터 구조에 대한 엄격한 검증이 부족합니다.
- **개선**:
    - **TypeScript**: `Zod`를 사용하여 Python에서 온 응답의 스키마를 검증하십시오.
    - **Python**: `Pydantic`을 사용하여 TS에서 온 요청의 유효성을 검사하십시오.
    - 공유된 스키마 정의를 통해 양측의 데이터 불일치 문제를 사전에 방지하십시오.

### 2.2 에러 전파(Error Propagation) 개선
- **현황**: Python에서 발생한 예외(Exception)가 TS 레이어에서 적절히 캡처되지 않거나, 사용자에게 단순히 "Error"로만 표시될 수 있습니다.
- **개선**: Python의 `traceback`을 구조화된 JSON 응답에 포함시키고, VS Code의 `window.showErrorMessage`를 통해 구체적인 에러 원인과 스택 트레이스를 (개발 모드에서) 확인할 수 있게 하십시오.

## 3. 핵심 로직 및 파싱 (Parsing & Logic)

### 3.1 정규표현식 기반 파싱의 한계 극복
- **현황**: `commandParser.ts`와 `labelDetection.ts` 등에서 복잡한 LaTeX 구조를 정규표현식으로 처리하고 있어, 중첩된 괄호나 복잡한 매크로 상황에서 취약할 수 있습니다.
- **개심**: 가능하다면 LaTeX AST 파서(예: `latex-utensils` 또는 `Arara` 관련 라이브러리)를 도입하여 구조적 분석을 수행하십시오.

### 3.2 Python 엔진 성능 최적화
- **현황**: 단일 Python 프로세스가 모든 계산(`calc`), 그래프(`plot`), 데이터 쿼리(`query`)를 처리하므로, 시간이 오래 걸리는 작업이 다른 요청을 차단할 가능성이 있습니다.
- **개선**: Python 측에서 `asyncio`를 활용하거나, 무거운 작업(예: ODE solving, 대량 데이터 쿼리)을 별도의 워커 스레드에서 실행하여 응답성을 높이십시오.

## 4. 테스트 및 품질 관리 (Testing & QA)

### 4.1 통합 테스트(Integration Tests) 강화
- **현황**: 현재 테스트는 각 모듈의 단위 테스트 위주입니다. TS와 Python 간의 실제 통신을 포함한 엔드투엔드(E2E) 테스트가 부족합니다.
- **개선**: 실제 Python 프로세스를 띄우고 명령어를 전달하여 결과를 검증하는 통합 테스트 시나리오를 추가하십시오.

### 4.2 CI/CD 환경 구축
- **현황**: GitHub Actions 등을 통한 자동화된 빌드 및 테스트 환경이 확인되지 않습니다.
- **개선**: `.github/workflows`를 추가하여 매 Push/PR마다 린트, 타입 체크, 테스트가 실행되도록 설정하십시오.

## 5. 기타 UX/기능 개선

### 5.1 증분 스캔(Incremental Scanning)
- **현황**: `labelDetection.ts` 등이 파일 전체를 매번 다시 스캔할 경우 대용량 LaTeX 파일에서 성능 저하가 우려됩니다.
- **개선**: 변경된 범위(Range)만 부분적으로 재스캔하는 로직을 검토하십시오.

### 5.2 의존성 관리
- **현황**: `python_backend/requirements.txt`에 버전이 명시되어 있지 않을 수 있습니다.
- **개선**: 라이브러리 버전을 고정(Pinning)하여 개발 환경과 사용자 환경의 일관성을 유지하십시오.

# TeX-Machina 프로젝트 개선 제안서 (fix.md)

이 문서는 TeX-Machina VS Code 확장 프로그램의 코드베이스를 분석하여 도출된 아키텍처, 성능, 및 유지보수 측면의 개선 사항을 정리한 것입니다.

## 1. 아키텍처 및 코드 구조 개선 (Extension Side)

### 1.1 `src/extension.ts` 리팩토링 (High Priority)
- **문제점**: 현재 `activate` 함수가 너무 비대하며(700라인 이상), 명령 등록, 웹뷰 관리, Python 프로세스 제어, 이벤트 리스너가 모두 섞여 있습니다.
- **개선안**: 책임을 분리하여 클래스화합니다.
  - `PythonBridge`: Python 프로세스 생명주기 관리 및 통신(JSON RPC 스타일) 담당.
  - `CommandManager`: 모든 VS Code 커맨드 등록 및 라우팅 담당.
  - `WebviewManager`: 웹뷰 프로바이더 등록 및 메시징 담당.
  - `StatusManager`: 전역 상태(현재 에디터, 선택 영역 등) 관리.

### 1.2 명령어 파서(`commandParser.ts`)의 견고화 (Medium Priority)
- **문제점**: 옵션 구분자(` /`)가 특정 공백 패턴에 의존하고 있어 사용자의 입력 실수에 취약합니다. 또한, `x > 0`과 같은 수식 내 비교 연산자가 명령어 구분자(`>`)와 충돌할 위험이 있습니다.
- **개선안**: 
  - 상태 머신 기반의 파서를 도입하여 문맥(Context)에 따른 구분자 해석 기능을 강화합니다.
  - 수식 내부의 비교 연산자를 보호하기 위한 룩어헤드(Look-ahead) 로직을 추가합니다.

### 1.3 전역 상태 관리 및 세션 격리 (Medium Priority)
- **문제점**: 여러 에디터 탭이 열려 있을 때 전역 변수(`currentEditor` 등) 공유로 인해 비동기 응답이 섞일 수 있습니다.
- **개선안**: 요청마다 고유한 `requestId`를 부여하고, 이를 기반으로 콜백을 매칭하는 `RequestRouter` 패턴을 도입합니다.

## 2. Python 백엔드 및 시각화 개선 (Backend Side)

### 2.1 시각화 데이터 관리 최적화 (High Priority)
- **문제점**: `plot_engine.py`가 생성하는 `.dat` 파일이 `data/` 폴더에 무한히 쌓여 프로젝트 용량을 차지합니다.
- **개선안**: 
  - `DataGarbageCollector`: 일정 시간(예: 1시간)이 지나거나 파일 개수가 임계치를 넘으면 오래된 `.dat` 파일을 자동 삭제하는 로직을 추가합니다.
  - 3D 그래프의 대용량 점 데이터를 JSON 문자열 대신 Base64 인코딩된 바이너리(TypedArray)로 전송하여 통신 오버헤드를 줄입니다.

### 2.2 LaTeX 전처리 및 SymPy 연동 안정성 (High Priority)
- **문제점**: 정규표현식 기반의 전처리가 복잡한 중첩 수식에서 한계를 보입니다.
- **개선안**: 
  - `latex2sympy2` 라이브러리의 AST를 직접 조작하여 더 정교한 수식 변환을 수행합니다.
  - `ode`, `pde` 엔진의 변수 감지 로직을 심볼릭 분석 중심으로 재편하여 정확도를 높입니다.

### 2.3 `calc` 기능의 모듈식 설계 및 확장성 강화 (High Priority)
- **문제점**: 새로운 계산 기능(미분, 적분, 행렬 등)을 추가할 때마다 `calc_engine.py`의 거대한 분기문(if-else)을 수정해야 하며, 유사한 전처리 로직이 중복됩니다.
- **개선안**:
  - **플러그인 레지스트리 패턴**: 데코레이터(`@register_calc`)를 사용하여 각 계산 기능을 독립된 파일(`calculators/derivative.py` 등)로 분리하고 동적으로 로드합니다.
  - **표준화된 실행 파이프라인**: 모든 계산 모듈이 `Pre-process -> Validate -> Compute -> Post-process`의 추상 베이스 클래스 인터페이스를 따르도록 설계합니다.
  - **메타데이터 기반 옵션 매핑**: 각 모듈이 지원하는 옵션 스키마를 정의하고, 이를 기반으로 TypeScript 프론트엔드와 연동하여 동적으로 명령어를 제안/검증합니다.
  - **수식 해석 엔진(Formula Engine) 공통화**: LaTeX 수식 파싱, 변수 추출 등 공통 로직을 독립된 유틸리티로 분리하여 모듈 간 재사용성을 극대화합니다.

## 3. 사용자 경험 및 기능 확장 (UX & Core Side)

### 3.1 수식 내비게이션(`nodeNavigation.ts`)의 지능화 (Medium Priority)
- **문제점**: 현재 점프 포인트가 텍스트 패턴 매칭에 의존하여 `\frac` 등의 복잡한 구조에서 논리적 이동이 부연스럽습니다.
- **개선안**: 수식의 계층 구조를 트리 형태로 파싱하여, "분자 -> 분모 -> 다음 항" 순으로 이동하는 시맨틱 내비게이션을 구현합니다.

### 3.2 기능 확장성 및 사용자 정의 (Low Priority)
- **문제점**: 단위 확장(`unitExpander.ts`) 및 다이어크리틱(`diacritics.ts`) 리스트가 하드코딩되어 있습니다.
- **개선안**: `package.json`의 `contributions`를 확장하여 사용자가 직접 커스텀 단위와 매크로를 등록할 수 있는 기능을 제공합니다.

### 3.3 라벨 종속성 분석 성능 개선 (Medium Priority)
- **문제점**: 대규모 프로젝트에서 모든 `.tex` 파일을 매번 전체 스캔하는 것은 비효율적입니다.
- **개선안**: 워크스페이스 내 파일들의 라벨 정보를 메모리에 캐싱하고, 수정된 파일만 다시 분석하는 증분 인덱싱(Incremental Indexing)을 도입합니다.

## 4. 테스트 및 품질 보증

### 4.1 회귀 테스트 자동화
- **문제점**: 복잡한 수학 연산의 결과가 SymPy 버전에 따라 달라질 수 있습니다.
- **개선안**: `python_backend/tests` 폴더를 정비하여 주요 수학 연산(ODE, Integral, Laplace 등)에 대한 표준 LaTeX 입출력 테스트 케이스를 100개 이상 확보합니다.

### 4.2 성능 모니터링
- **문제점**: Python 프로세스의 응답 속도가 사용자 체감 품질에 큰 영향을 미칩니다.
- **개선안**: 각 명령의 실행 시간을 측정하여 임계값을 넘을 경우 사용자에게 진행 상태(Progress Bar)를 표시하는 UX를 추가합니다.
